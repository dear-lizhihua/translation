模块思维方式

正如前言中所讨论的，当我们正在开发软件项目时，复杂性似乎就在我们身边。抽象也是如此，它们将复杂性隐藏在我们不敢触及的礁石之下。这些复杂和抽象构成的礁石，恰是我们与此世界其他部分的接口，因此我们几乎无法思考它。在 JavaScript 中也不例外。相反，由于动态语言的强大，在我们使用它们时编写复杂程序也更容易，甚至很诱人。

首先，让我们讨论如何更好地将抽象、接口，以及它们的基本概念，应用于我们的工作。然后，我们可以最大限度地减少在处理项目、特性、部分功能时需要注视的复杂程度，而分解为一个个功能分支。

1.1 模块化思维简介

拥抱模块化思维是可以理解的，是因为一切最终都会不可避免地涉及到复杂度。与此同时，虽然这种复杂度可以扫除到一个接口中，几乎不会再被人看到或联想到。但是 - 这是一个棘手的部分 - 接口需要精心设计，以使使用它的用户不会因为粗糙的接口而感到沮丧。这种挫败感甚至可能导致我们深入了解底层原理，并且发现它的实现要比让我们感到沮丧的糟糕接口复杂得多，并且如果接口首先不存在，在可维护性和可读性方面，我们会做得更好。

系统可以精细组织：我们可以将它们分成多个应用程序组成的项目，包含多个应用程序级别的层，其中我们可以拥有数百个模块，由数千个函数组成，等等。粒状方式可以帮助我们编写易于理解和维护的代码，通过实现合理程度的模块化，以及保持我们的理智。第1.4节“模块化粒度”中，我们将讨论如何最好地利用这种粒度来创建模块化应用程序。

每当我们描绘一个组件时，就会有一个公共接口，系统的其他部分可以利用它来访问我们的组件。接口或 API 包含组件公开的方法或属性集。这些方法或属性也可以称为接触点 - 可以与其公开交互的接口的方面。接口的接触点越少，其表面积越小，接口越简单。具有大表面积的接口非常灵活，但考虑到接口暴露的大量功能，可能也更难理解和使用。

接口具有双重用途。它允许我们开发组件的新部分，仅暴露出为公共消费所准备的功能，同时保留不打算与其他组件共享的私有内容。同时，它允许消费者 - 利用我们的接口的组件或系统 - 从我们开放的功能中的获得好处，而不关心我们如何实现该功能的细节。

健壮、文档化的接口是隔离复杂代码的最佳方式之一，以便其他人可以在不知道任何实现细节的情况下使用其功能。可以将具有健壮接口的系统装置累积起来以形成层，例如企业应用程序中的服务或数据层。通过这样做，我们可以在很大程度上隔离和限制逻辑到其中一个层，同时将表现层关注点与严格的业务相关或持久性相关关注点分离。这种强制分离是有效的，因为它使各个组件保持整洁和分层一致。由模式或形状相似的组件组成的统一层提供了一种熟悉感，从单个开发人员的角度来看，他们可以在此持续基础上更直接地消费它们。随着时间的推移，这些开发人员越来越习惯熟悉的 API 形状。

由于难以提供足够的接口设计，依靠一致的 API 形状是提高生产率的好方法。当我们始终如一地利用类似的 API 形状时，我们不必每次都提出新的设计，消费者可以放心相信你没有每次都重新发明轮子。我们将在接下来的章节中详细讨论 API 设计。

1.2 模块化简史

谈到 JavaScript，模块化是一个现代概念。在本节中，我们将快速回顾并总结 JavaScript 世界中模块化演变的里程碑。无论如何，本节并没有全面列出，而是说明 JavaScript 历史上的主要范式变化。

1.2.1 脚本标记和闭包

早期，直接在 HTML <script> 标签中内联 JavaScript。后来，它被放置到专用脚本文件，所有脚本文件都共享一个全局作用域。

在这些文件或内联脚本之一中声明的任何变量或绑定，都会在全局 window 对象上留下痕迹。这会在完全不相关联的脚本中产生泄漏，这些泄漏脚本可能会导致冲突甚至破坏体验，因为一个脚本中的变量，可能会无意中替换另一个脚本所依赖的全局变量：

<script>
  var initialized = false

  if (!initialized) {
    init()
  }

  function init() {
    initialized = true
    console.log('init')
  }
</script>

<script>
  if (initialized) {
    console.log('was initialized!')
  }

  // even `init` has been implicitly made a global variable
  console.log('init' in window)
</script>

最终，随着 web 应用程序的规模和复杂性不断增长，作用域的概念和全局作用域带来的危险，变得显然易见而且为人所知。立即调用函数表达式（Immediately Invoked Function Expressions, IIFE）被发明并成为当下的主流。 IIFE 通过将整个文件或文件的一部分，封装在取值后立即执行的函数中来工作。JavaScript 中的每个函数都创建了一个新的作用域，即 var 变量绑定将包裹在 IIFE 中。尽管变量声明被提升到其所在作用域的顶部，但由于 IIFE 容器包裹，它们永远不会成为隐式全局变量，从而抑制了隐式 JavaScript 全局变量造成的脆弱性。

在下面的示例片段中可以找到 IIFE 几种风格。每个 IIFE 中的代码都是隔离的，只能通过类似 window.fromIIFE = true 的显式语句，将变量逃逸全局上下文中。

(function() {
  console.log('IIFE using parenthesis')
})()

~function() {
  console.log('IIFE using a bitwise operator')
}()

void function() {
  console.log('IIFE using the void operator')
}()

使用 IIFE 模式，library 通常通过暴露来创建模块，然后在 window 对象上复用单个绑定，从而最小化全局命名空间污染。下面片段展示了我们如何在这些基于 IIFE 的库中，创建一个具有 sum 方法的 mathlib 组件。如果我们想要向 mathlib 添加更多模块，我们可以将它们每个都放在一个单独的 IIFE 中，它将自己的方法添加到 mathlib 公共接口，而其他定义新功能部分的方法，对于此组件都可以保持私有：

void function() {
  window.mathlib = window.mathlib || {}
  window.mathlib.sum = sum

  function sum(...values) {
    return values.reduce((a, b) => a + b, 0)
  }
}()

mathlib.sum(1, 2, 3)
// <- 6

巧合的是，这种模式是对 JavaScript 工具兴起热潮的一次公开邀请，第一次允许开发人员将每个 IIFE 模块拼接到一个文件中，从而减轻了网络压力。只需要依托现有基本打包方案，就能解决自动分号插入，并且在不会破坏应用程序逻辑的情况下减少内容。

IIFE 方法的问题在于没有明确的依赖树。开发人员必须以精确的顺序制作组件文件列表，以便在依赖于它们的任何模块递归之前加载依赖关系。

1.2.2 RequireJS, AngularJS 和依赖注入

这是自模块系统（例如 RequireJS 或 AngularJS 中的依赖注入机制，这两个系统都允许我们明确命名每个模块的依赖）出现以来，我们几乎没有考虑过的问题。

下面示例展示了如何使用 RequireJS 的 define 函数，定义 mathlib/sum.js library，此函数会被添加到全局作用域。然后，来自define callback 的返回值将作为模块的公共接口：

define(function() {
  return sum

  function sum(...values) {
    return values.reduce((a, b) => a + b, 0)
  }
})

然后我们可以有一个 mathlib.js 模块，它聚合我们想要包含在 library 中的所有功能。在我们的示例中，只依赖于 mathlib/sum，但我们可以以同样的方式，列出许多我们需要用到的模块。我们通过在数组中使用它们的路径来列出每个依赖，并且我们以同样的顺序，将它们的公共接口作为参数传递到我们的 callback 回调中：

define(['mathlib/sum'], function(sum) {
  return { sum }
})

现在我们已经定义了一个 library ，我们可以使用 require 来消费它。请注意在此代码片段中，是如何为我们解析依赖链的：

require(['mathlib'], function(mathlib) {
  mathlib.sum(1, 2, 3)
  // <- 6
})

这是 RequireJS 和它与生俱来的依赖树的优势。无论我们的应用程序是否包含几百或上千个模块，RequireJS 都会解析依赖关系树，而无需费心维护模块列表。鉴于我们已经将依赖关系准确地列在需要它们的位置，我们已经消除了每个组件的长列表及其相互关联的必要性，以及维护这样的列表的容易出错的过程。消除这么大的复杂性来源仅仅是副作用，而不是主要的好处。

在模块级别，依赖性声明中的这种显式性使得组件与应用程序的其他部分之间的关系变得明显。 反过来，这种明确性促进了更大程度的模块化，这是因为跟随依赖链有多么困难，之前是无效的。

RequireJS 并非没有问题。整个模式围绕其异步加载模块的能力展开，由于其执行效果不佳，因此不建议在生产部署中使用。使用此异步加载机制，在许多代码执行前，你需要以瀑布形式发出数百个网络请求。必须使用其他工具来优化生产版本。其次是冗长性因素，最终会产生一长串依赖关系、一个 RequireJS 函数调用、还有模块的回调。关于这一点，有很多不同的 RequireJS 函数以及调用这些函数的几种方法，使它的用法变得复杂起来。但这种 API 不是最直观的，因为完成相同操作的方式有很多：比如声明一个具有依赖性的模块。

AngularJS 中的依赖注入系统也遇到许多相同的问题。这是当时一个很好的解决方案，它依靠智能字符串解析来避免依赖数组，使用函数参数名称来解析依赖。这种机制无法与压缩工具兼容，压缩工具会将参数重命名为单个字符，从而破坏了注入器。

在 AngularJS v1 整个生命周期的后期，会引入一个构建任务，该任务将如下代码转换：

module.factory('calculator', function(mathlib) {
  // …
})

转换为如下格式的代码，这样做是安全的，因为它包含显式依赖列表。

module.factory('calculator', ['mathlib', function(mathlib) {
  // …
}])

不需要说，引入这个鲜为人知的构建工具所耽搁的时间，再加上增加额外的构建步骤导致的过度工程化，破坏了本不该破坏的东西，因此不鼓励使用这种模式，其带来的好处微不足道。开发人员大多选择坚持使用他们熟悉的类似 RequireJS 的硬编码依赖数组格式。

1.2.3 Node.js 和 CommonJS 到来

Node.js 欢呼雀跃的众多创新之一是 CommonJS 模块系统，简称 CJS。利用这一事实，Node.js 程序可以访问文件系统。CommonJS 标准更加符合传统的模块加载机制。在 CommonJS 中，每个文件都是一个具有自己的作用域和上下文的模块。使用同步的 require 函数来加载依赖关系，该函数可以在模块的生命周期中的任何时间动态调用，如以下代码段所示：

const mathlib = require('./mathlib')

与 RequireJS 和 AngularJS 一样，CommonJS 依赖项也由路径名引用。主要区别在于样板函数(boilerplate function)和依赖项数组(dependency array)现在都消失了，并且可以将模块接口分配到一个变量绑定(variable binding)上，或在可以任何地方使用 JavaScript 表达式引用。

与 RequireJS 或 AngularJS 不同，CommonJS 非常严格。在 RequireJS 和 AngularJS 中，每个文件可以有许多动态定义的模块，而 CommonJS 在文件和模块之间是一对一的映射。同时，RequireJS 有几种声明模块的方法，而 AngularJS 除了它的依赖注入机制与 AngularJS 框架本身紧密耦合的事实，还具有工厂(factory)、服务(service)、提供者(provider)等。相反，CommonJS 只是一种声明模块的方式。任何 JavaScript 文件都是模块，调用 require 会加载依赖项，分配给 module.exports 的任何内容都是其接口。这样可以实现更好的工具和代码自检，使工具更容易理解 CommonJS 组件系统的层次结构。

为弥合 Node.js 服务器和浏览器的 CommonJS 模块之间的鸿沟，最终发明出 Browserify。使用 browserify CLI 程序，并为它提供入口起点模块的路径，可以将无法想象数量的模块，组合到为浏览器准备的单独 bundle 中。CommonJS 的杀手级功能，即 npm package registry，在帮助其接管模块加载生态系统方面起着决定性作用。

当然，npm 不仅限于 CommonJS 模块甚至 JavaScript 包，但这在很大程度上仍然是其主要用例。只需轻按几下即可为你的 Web 应用程序中提供数千个 package（现在已超过50万个，并且正在稳定增长）的广阔前景，以及能够在 Node.js web server 和每个客户端的 web browser 中重用系统的大部分功能的能力，这种竞争优势对于其他系统来说都无法赶上。

1.2.4 ES6, import, Babel 和 webpack

随着 ES6 在 2015 年 6 月标准化，并且 Babel 在此之前很久就可以将 ES6 转译成 ES5，一场新的革命正在迅速来临。ES6 规范引入 JavaScript 原生自带的模块语法，通常称为 ECMAScript 模块（ESM - ECMAScript Modules）。

ESM 在很大程度上受到 CJS 及其前身的影响，提供了静态声明 API 和基于 promise 的动态可编程 API，如下所示。

import mathlib from './mathlib'
import('./mathlib').then(mathlib => {
  // …
})

同样在 ESM 中，每个文件都是具有自己的作用域和上下文的模块。与 CJS 相比，ESM 的一个主要优点是 ESM 使用（并且鼓励使用）静态导入依赖的方式。静态导入极大地提高了模块系统的自检能力，因为可以从系统中每个模块的抽象语法树（AST）进行静态分析和词法分析来提取它们。ESM 中的静态导入被限制在模块的最顶层级别，从而进一步简化了解析和自检。与 CommonJS require() 相比， ESM 的另一个优点是 ESM 指定了一种执行异步模块加载的方式，这意味着可以根据特定事件，而按需进行并发或延迟加载应用程序依赖图的某些部分。尽管在撰写本文时，该功能尚未在大多数环境中实现，但是，有强烈迹象表明[1] Node.js 将来会合并它。

在 Node.js v8.5.0 中，在加入 --experimental-modules flag 之后可以支持 ESM，需要让模块使用 .mjs 文件扩展名。大多数常青浏览器已经无需额外添加 flag 就可以支持 ESM。

webpack 是 Browserify 的继任者，它由于具有更广泛的功能，而在很大程度上取代了通用模块打包工具的角色。就像 Babel 和 ES6 一样，webpack 长期以来一直支持 ESM，包括静态的 import 和 export 语句和动态的 import() 类似函数表达式。它在采用 ESM 方面取得了极其有效的成果，这在很大程度上要归功于引入了 "code-splitting(代码拆分)" 机制[2]，此机制可以将应用程序拆分为不同的 bundle，以提高首次加载体验时的性能。

鉴于 ESM 是 JavaScript 语言的原生语法（和 CJS 相反），它有望在几年后完全占据模块生态系统。

1.3 模块化设计的优势

我们已经解决了这样一个事实：与单个共享的全局作用域相比，模块化有助于避免因各种模块的作用域多样化而在各种名称中出现意外冲突。除了冲突的修复之外，跨文件的模块化，限制了我们在处理任何一个特定功能时必须注意的复杂程度。因此，我们的团队能够专注于手头的任务并提高工作效率。

由于这点，可维护性或在代码库中实现修改的能力也显着提高。当代码简单且模块化时，它更容易构建和扩展。无论团队规模如何，可维护性都是有价值的：即使在一个团队中，如果我们在一开始就认真考虑编写可维护代码，将一段代码保持原状几个月然后再回到它，那么可能很难改进甚至理解。

模块化代码默认是高度可维护的。通过保持代码片段简单并遵循单一责任原则（SRP），每个代码目标只实现一个目的，并将这些简单的代码片段组合成更复杂的组件，我们就能够构建更大的组件，并最终组合成整个应用程序。当程序中的每段代码都是模块化的时，在我们查看单个组件时代码库会看起来很简单，但总的来说，它能够表现出复杂的行为，就像我们在本章开始的图书出版过程中讨论的那样。

模块化应用程序中的组件由其接口定义。这些组件的本质不是它们的实现，而是它们的接口。当接口设计良好时，它们可以以不间断的方式增长，增加它们可以满足的用例数量，而不会影响现有的使用。当我们有一个精心设计的接口时，该接口背后的实现变得易于调整或完全置换。强接口可以有效地隐藏弱实现，如果接口能提供支撑，可以在以后重构为更健壮的实现。强大的接口也非常适合单元测试，因为我们不必担心实现，并且可以测试接口 - 组件或函数的输入和输出。如果接口可良好测试并且健壮，我们肯定可以考虑在次级平面上实现这些接口。

鉴于首要需求是具有直观的接口（没有与其实现相互耦合），这些实现是次要的，我们可以关注灵活性和简单性之间的权衡。灵活性不可避免地以增加复杂性为代价，这是不提供灵活接口的良好借口。同时，灵活性通常是必要的，因此我们需要通过决定在接口中消除多少僵化程度来达到适当的平衡。这种平衡意味着接口因其易用性而吸引消费者，但它还可以在需要时实现高级或更常见的用例，而不会对易用性造成太大的不利影响，或者以极大强化实现的复杂程度为代价。

我们将在接下来的几章中讨论灵活性、简单性、可组合性和适当数量的未来验证之间的权衡。

1.4 模块化粒度

我们可以在给定系统的每个级别上应用模块化设计概念。如果一个项目的需求超过其初始范围，我们可能会考虑将该项目拆分为几个较小的项目，其中较小的团队更易于管理。应用程序也是如此：当它们变得庞大或足够复杂时，我们可能希望将它们分成不同的产品。

当我们想要使应用程序更易于维护时，我们应该考虑创建明确定义的代码层，以便我们可以水平增长每个层，同时防止这些添加的复杂性扩散到其他不相关的层。相同的思维过程可以应用于单个组件，将它们分成两个或更多个较小的组件，然后由另一个小组件连接在一起，这些组件可以充当组合层，其唯一的责任是将几个底层组件编织在一起。

在模块级别上，我们应该努力使函数简单且易于表达，并使用描述性名称而且不要承担太多责任。在特定的异步流下，也许我们会有一个专门用于将一组任务组合在一起的函数，同时为控制流中执行的每个任务提供一些其他函数。最高级别的流控制函数，可以作为模块的公共接口方法暴露，但是其中应视为公共接口的唯一部分是，函数从输入接收到的那些参数，以及该同一个顶级函数产生的输出。其他所有内容都视为实现细节，由此被认为是可交换的。

模块的内部功能也不会像接口那样严格：只要拥有公共接口，我们就可以修改实现 - 如果需要的话，我们还可以修改包括构成实现的那些函数接口。但是，这并不是说我们应该故意减少处理这些接口。正确的模块化设计的关键是对所有接口都具有最大的尊重，包括暴露出内部函数的接口。

在函数中，我们还将注意到有必要组件化实现的各个切面，以函数调用的方式为这些切面命名，从而推迟不需要在函数主体中立即处理的复杂逻辑，直到稍后再通读给定的一段代码。我们正在编写旨在让其他人甚至自己将来都能阅读和书写的程序。几乎每个做过大量编程工作的人，在浏览几个月前他们自己编写的一段代码时都会感到沮丧，后来才意识到，他们是用新的眼光回顾，然后他们提出的设计，并没有达到他们最初的预期。

请记住，计算机程序开发在很大程度上是人类和协作的努力。我们并未特意针对计算机进行优化，以使其达到尽可能快地运行程序。如果我们真这样做的话，就要在电路板上编写二进制或硬编码逻辑。相反，我们的工作重点是增强组织的能力，使其开发人员可以保持生产力，并能够快速理解甚至修改他们以前从未遇到过的代码。在惯例和实践的轻微簇拥下进行工作-使开发人员处于平滑过渡的状态-通过确保将来的开发与应用程序目前状态保持一致，来产生开发闭环。

回到性能，我们应该将其视为一项功能，在大多数情况下，我们对它的重视程度并不比对其他功能要高。除非出于业务原因而将性能作为我们系统的一个定义功能，否则我们不必考虑确保系统在所有代码路径上均以最高速度运行。这样做必然会产生难以维护、调试、扩展和解释的高度复杂的应用程序。

作为开发人员，我们经常也会过度架构，有关性能优化的许多理由也适用于此。当我们扩展到每秒数十亿次事务时，设计出一种无所不包的体系结构，可能会为我们减少很多麻烦，但这也会花费我们大量的前期时间，并且使我们陷入一系列难以跟上的抽象过程，在短期内没有可预见的收益。当我们专注于已经遇到或可能即将遇到的问题时，而不是试图在没有任何数据的情况下，规划基础设施和吞吐量的曲棍球式增长，来支持我们所期待的曲棍球式增长，这种情况会好得多。

当我们没有长期计划时，就会发生一件有趣的事情：我们的系统更加自然地增长，可以适应近期的需求，并逐渐朝着支持更大应用程序和更多需求集合方向发展。当这种进展是渐进式的，我们就要注意随着增长，应当采取选择抽象或丢弃抽象的纠正措施。如果我们过早地确定抽象，而最终导致它们成为错误的抽象，那么我们将为此付出沉重的代价。错误的抽象迫使我们将整个应用程序屈服于它们的意愿，一旦我们意识到抽象是不好的并且应该被删除，我们可能会投入大量的精力，以至于退出可能也会付出高昂的代价。而我们很想保留的抽象，仅仅是因为我们花了很多时间、汗水和鲜血，伴随着这些沉没成本误区，这确实显得很危险。

我们将在本书的重要部分专门介绍，如何在合适的时间，识别和利用正确的抽象，从而最大程度地降低它们带来的风险。

1.5 模块化 JavaScript：必要性

由于其历史，JavaScript 在模块化设计方面特别有趣。在网络的早期和很长一段时间内，没有任何既定的惯例，除了显示警报框外，很少有人知道这门语言。作为一种尚未高度成熟的高度动态语言，JavaScript 在静态类型语言（如 Java 或 C#）和更加常用的动态语言（如 Python 或 PHP）之间处于一个奇怪的位置。

web 缺乏原生模块化（可以通过 HTML <script> 标签加载方式，将程序进行分块加载），与其他执行环境形成了鲜明对比，在其他执行环境中，程序可以由任意数量的文件组成，并且语言层面原生支持模块化体系结构，并且编译器和文件系统都基于编程环境。而在 web，我们才刚刚开始接触原生模块的表面，这些其他编程环境自诞生以来就拥有的东西。正如我们在 1.2 节中讨论的那样，缺少原生模块加载机制，再加上缺少原生模块（仅仅只有共享全局作用域的文件），这些历史原因迫使 web 社区在实现其模块化方式上变得富有创意。

JavaScript 原生模块规范最终能够成为语言级别规范，在很大程度上受到 JavaScript 社区主导的努力所影响。即使在撰写本文时，距离在 web 中有效使用原生模块系统大概还有 2 到 3 年的时间。 web 这种缺陷可以通过在其他环境普遍采用的模式来证明（例如分层或基于组件的体系结构），而这是在 web 漫长时间都未曾遇到的。

很少有人真正地将 JavaScript 视为现代开发平台，直到 2004 年 4 月推出 Gmail Beta 客户端，向大家演示了通过异步 JavaScript HTTP 请求，提供单页面应用程序体验的强大能力，然后在 2006 年首次发布了 jQuery，从而提供了无烦恼的跨浏览器 web 开发经验。

随着诸如 Backbone, Angular, Ember 和 React 之类的框架的出现，新技术和突破也使 web 出现了增长。在 ES6+ 版本下编写代码，然后将其部分代码转换为 ES5，以获得更广泛的浏览器支持；同构渲染，通过在 server 和 client 上的复用相同代码，实现初始页面加载时快速呈现页面，以及实现导航时快速加载页面；自动代码打包，将组成应用程序的模块打包到单个 bundle 中，以优化交付；根据路由进行 bundle-splitting，最终输出多个 bundle，每个 bundle 都是为初始访问路由的速度所优化；在 JavaScript 模块级别打包 CSS，因此 CSS（本身不具有原生模块语法）也可以拆分为多个 bundle；在编译时优化资源（例如图像），提高开发效率，同时保证生产部署高效，这无数种方式，都是 web 中创新迭代过程的一部分。

创新的爆炸式增长不仅源于纯粹的创造力，还源于必要性：由于 web 的使用范围越来越多、目的和需求越来越多，web 应用程序变得越来越复杂。因而从逻辑上讲，围绕的生态系统也需要不断增长，以适应这些扩展的需求，在更好的工具、更好的库、更好的编码实践&体系结构&标准&模式这些方面，通常也会有更多选择。

在下一章中，我们将复杂性的含义进行分解，并在编写程序时，会开始对复杂度设立防御措施。通过遵循一些跨组件层面封装逻辑的规则，我们将开始更简单的程序设计的旅程。
